# Costing Journal - 2026-03-17

## Goal

Figure out a provider-agnostic way to store model pricing history so Mastra can turn token usage metrics into cost metrics.

Primary target:

- Join pricing data with the token metrics emitted in [observability/mastra/src/metrics/auto-extract.ts](/Users/epinzur/src/github.com/mastra-ai/mastra/observability/mastra/src/metrics/auto-extract.ts).

Product framing:

- This system is for cost estimates, not invoice-grade billing.
- It is acceptable to start with approximate prices when runtime metadata is incomplete.
- The system should improve over time as pricing inputs and observability context improve.

This is enough for base token pricing, but not enough for several real pricing branches found in the provider docs in this folder.

## Pricing Dimensions Seen In The Source Docs

The collected provider docs already show recurring dimensions that do not fit a simple `provider + model + input/output` table:

- Service tier: `standard`, `batch`, `priority`, `flex`, `fast`
- Cache variants: cache read, cache write, cache TTL-dependent write pricing
- Region or data residency uplifts
- Long-context thresholds where the entire request changes price after an input-token boundary
- Modality-specific token pricing: text, audio, image, reasoning
- Non-token billable units: images, video seconds, tool invocations, search queries, voice minutes, container sessions

Conclusion: we need a generic pricing-rule table, not just a token-price lookup table.

## Storage Direction

Parquet looks right for the normalized historical dataset because:

- it is columnar and compact
- it will query well for time-series analysis and backfills
- it is a good fit for batch joins against usage exports

Parquet alone is a poor authoring format for a git repo, so the working model should probably be:

1. Keep raw provider snapshots in human-readable formats (`.md`, `.json`, `.txt`) as source evidence.
2. Parse them into a normalized text artifact that is easy to diff in git, likely JSONL.
3. Materialize Parquet from that normalized layer for analytics and downstream jobs.

Working assumption: JSONL is the review format, Parquet is the storage/query format.

## Product Requirements Added Today

The intended product shape is now clearer:

- Mastra should expose cost estimates from usage data using best-effort pricing joins.
- There should be a hosted API for centrally updated pricing estimates.
- There should also be a local copy of the pricing dataset for offline or low-latency use.
- The package should ship with a recent embedded snapshot so it works out of the box.
- Over time, users should be able to refresh or recalculate estimates using newer pricing data.

This pushes the design toward a snapshot-based pricing package plus an optional remote sync layer.

## Research: Langfuse and Braintrust

I checked whether existing observability products document how they calculate LLM usage cost.

### Langfuse

Langfuse publicly documents a fairly complete cost-tracking approach.

Key behaviors documented by Langfuse:

- usage and cost can either be ingested directly or inferred from the `model` field
- ingested usage/cost takes priority over inferred values
- inferred pricing is calculated at ingestion time using the model definition available at that time
- model definitions contain prices per usage type, and usage-detail keys must match those price keys exactly
- custom model definitions are supported and can be managed via UI or API
- model matching is regex-based via `match_pattern`
- user-defined model definitions override Langfuse-maintained definitions
- tiered pricing is supported with ordered conditions, for example `input > 200000`
- reasoning-model cost cannot be inferred correctly without explicit token usage
- changing a model definition only affects new generations, not previously ingested ones

What is especially relevant for our design:

- Langfuse separates `usage_details` from `cost_details`
- it supports arbitrary usage keys instead of a hardcoded small enum
- it handles threshold pricing explicitly
- it treats cost inference as time-sensitive at ingestion time

That last point differs from our direction. We want immutable price history so historical usage can be recomputed later against the correct effective rule interval, including after pricing corrections.

Useful ideas to borrow from Langfuse:

- exact usage-key to price-key matching
- explicit tier evaluation rules
- regex-based model matching for custom definitions
- support for direct ingestion of provider-returned cost when available

### Braintrust

Braintrust publicly documents cost tracking, but not with the same level of detail.

What Braintrust clearly documents:

- traces and spans capture token usage and cost in observability
- experiment views show `Estimated cost`, described as approximate cost based on pricing
- custom AI providers can be configured with:
  - input cost per million input tokens
  - output cost per million output tokens
  - input cache read cost per million cached input tokens read
  - input cache write cost per million input tokens written to cache
- those custom-provider cost fields are explicitly described as being used for experiment cost estimation

What I could infer, but not verify in equivalent detail:

- Braintrust likely estimates cost as token counts multiplied by configured per-million-token rates
- it likely supports cache-aware estimation when the relevant counts are present
- it likely has provider/model defaults internally for wrapped providers

What I could not find publicly:

- a detailed public description of tier matching or threshold pricing
- a documented history model with effective dates
- a public explanation of how corrections or retroactive price changes are handled
- a public source-of-truth model catalog comparable to Langfuse’s documented model-definition system

Working conclusion:

- Langfuse is a strong reference for ingestion-time inference, model-definition matching, and tiered pricing
- Braintrust is a useful reference for keeping the user-facing product honest by calling the result an estimate and by exposing configurable input/output/cache rates
- our design should combine those strengths while preserving explicit historical price intervals for later recomputation

Reference sources reviewed:

- Langfuse FAQ: `https://langfuse.com/faq/all/costs-tokens-langfuse`
- Langfuse docs: `https://langfuse.com/docs/observability/features/token-and-cost-tracking`
- Braintrust custom providers: `https://www.braintrust.dev/docs/integrations/ai-providers/custom`
- Braintrust instrumentation: `https://www.braintrust.dev/docs/instrument`
- Braintrust experiment results: `https://www.braintrust.dev/docs/guides/experiments/interpret`

## Research: LangSmith and Arize Phoenix

I also checked LangSmith and Arize Phoenix to see how they handle cost estimation.

### LangSmith

LangSmith publicly documents a detailed cost-tracking model that is close to Langfuse in several ways.

What LangSmith clearly documents:

- cost can be tracked either automatically from token counts plus model prices, or manually by sending explicit costs
- automatic costing requires token counts, provider, model, and a model pricing map
- token usage is broken down into input, output, and other categories
- input and output token details support subtype keys such as `cache_read`, `audio`, `image`, `reasoning`
- cost is computed greedily from most-specific token subtype to less-specific default prices
- custom model price entries support:
  - input and output base prices
  - input and output price breakdowns by token subtype
  - regex `Match Pattern`
  - provider matching
  - `Model Activation Date`
- if pricing is non-linear, LangSmith recommends computing costs client-side and sending explicit cost values
- LangSmith explicitly says pricing-map updates are not reflected in already logged traces and that it does not currently support backfilling pricing changes

What this suggests about their model:

- they support a richer token-cost mapping than a simple input/output pair
- they support activation-date-aware pricing selection for new traces
- they optimize for ingestion-time calculation, not later historical recomputation

Useful ideas to borrow from LangSmith:

- greedy subtype-first cost computation
- support for direct cost ingestion when pricing is non-linear
- model activation dates
- explicit distinction between automatic token-derived costs and manually supplied costs

Important difference from our direction:

- LangSmith does not currently backfill pricing changes for older traces
- we want to preserve historical rule intervals so the same usage can be recomputed later against corrected or newer pricing history

### Arize Phoenix

Arize Phoenix also documents a meaningful amount of cost-tracking behavior.

What Phoenix clearly documents:

- token-based costs are calculated automatically from token counts and model pricing data
- Phoenix uses OpenInference semantic conventions for token counts and model metadata
- required attributes include prompt/completion/total token counts plus `llm.model_name` and `llm.provider`
- optional detailed token counts include:
  - prompt `cache_read`
  - prompt `cache_write`
  - prompt `audio`
  - completion `reasoning`
  - completion `audio`
- Phoenix ships with a built-in model pricing table covering several major providers
- users can manage pricing in the UI and add custom models or override existing ones
- custom model pricing supports:
  - regex model-name matching
  - optional provider matching
  - prompt and completion base prices
  - `Start Date`
- advanced pricing supports detailed prompt/completion price breakdowns for cache, audio, image, video, and reasoning token types
- computed costs are surfaced at trace, span, session, experiment, and project levels

What I did not find documented as clearly:

- whether pricing changes are backfilled to historical traces
- whether Phoenix keeps immutable historical price versions internally
- how complex non-linear threshold pricing is represented beyond token-type breakdowns

Working conclusion:

- LangSmith is a strong reference for subtype-aware token costing and for explicitly falling back to direct cost ingestion for non-linear models
- Phoenix is a strong reference for aligning with OpenInference attributes and for configurable model pricing with start dates and token-type breakdowns
- both systems still appear more ingestion-oriented than history-oriented
- our design should keep their flexibility, but retain explicit historical intervals and correction events so recomputation remains possible

Additional reference sources reviewed:

- LangSmith cost tracking: `https://docs.langchain.com/langsmith/cost-tracking`
- LangSmith tracing guide: `https://docs.langchain.com/langsmith/log-llm-trace`
- Phoenix cost tracking: `https://arize.com/docs/phoenix/tracing/how-to-tracing/cost-tracking`

## Proposed Canonical Schema

One row should mean:

> one effective price for one billable meter under one pricing context

The schema should be split into layers, not forced into one file format:

1. source metadata
2. normalized pricing rules
3. published base/delta snapshot metadata
4. runtime estimate output

### Canonical rule schema v1

This is the core record we should normalize everything into.

Required fields:

| Field                | Type   | Notes                                                                 |
| -------------------- | ------ | --------------------------------------------------------------------- |
| `rule_id`            | string | Stable identifier for the logical pricing rule                        |
| `provider`           | string | Canonical provider id                                                 |
| `model_match_type`   | enum   | `exact`, `regex`, `family`, `alias`                                   |
| `model_match_value`  | string | Exact model id or pattern used to match                               |
| `meter`              | string | Canonical billed meter                                                |
| `unit`               | enum   | `token`, `image`, `second`, `minute`, `session`, `query`, `character` |
| `price_usd_per_unit` | number | Normalized numeric USD price                                          |
| `pricing_mode`       | enum   | `base`, `tier`, `modifier`, `tool`, `session`, `storage`              |
| `effective_start`    | string | ISO timestamp or date                                                 |
| `source_type`        | enum   | `provider`, `aggregator`, `derived`, `manual`                         |
| `source_name`        | string | Short source id like `langfuse`, `anthropic-docs`                     |
| `source_location`    | string | URL or local path used to derive the rule                             |
| `snapshot_date`      | string | Capture date for the source snapshot                                  |
| `content_hash`       | string | Hash of normalized content for diffing                                |

Optional fields:

| Field                | Type     | Notes                                                    |
| -------------------- | -------- | -------------------------------------------------------- |
| `currency`           | string   | Default `USD`                                            |
| `model_display_name` | string   | Source display name                                      |
| `model_family`       | string   | Optional grouping                                        |
| `provider_model_id`  | string   | Provider-specific model id if different                  |
| `service_tier`       | string   | `standard`, `batch`, `priority`, `flex`, `fast`, etc.    |
| `region`             | string   | Billing region or residency scope                        |
| `cache_role`         | enum     | `read`, `write`                                          |
| `cache_ttl`          | string   | `5m`, `1h`, etc.                                         |
| `threshold_basis`    | string   | e.g. `input_tokens_total`                                |
| `threshold_min`      | number   | Inclusive lower bound                                    |
| `threshold_max`      | number   | Exclusive upper bound                                    |
| `effective_end`      | string   | End of validity interval                                 |
| `source_confidence`  | enum     | `high`, `medium`, `low`                                  |
| `supersedes_rule_id` | string   | Prior rule logically replaced                            |
| `change_type`        | enum     | `insert`, `close`, `correct`, rare `delete`              |
| `notes`              | string   | Human parse notes or caveats                             |
| `raw_meter_keys`     | string[] | Source-native keys that mapped here                      |
| `tags`               | string[] | Freeform flags like `deprecated`, `estimated`, `preview` |

Recommended rule identity:

- `rule_id` should identify the logical pricing lane, not a specific versioned row
- price changes over time should create a new row with the same logical identity dimensions but a new effective interval
- one practical identity basis is:
  - `provider`
  - `model_match_type`
  - `model_match_value`
  - `meter`
  - `service_tier`
  - `region`
  - `cache_role`
  - `cache_ttl`
  - `threshold_basis`
  - `threshold_min`
  - `threshold_max`

## Fit With Current Metrics

There is a clean near-term path for standard token pricing:

- map each emitted usage metric to a canonical `meter`
- join on `provider + model + meter + snapshot`
- multiply usage by `price_usd_per_unit`

That would cover straightforward cases like:

- OpenAI standard text input/output pricing
- Anthropic base input/output/cache pricing
- Providers with explicit audio/image token rates

Because this is estimate-oriented, we can also define explicit fallback behavior:

- If a rule needs missing runtime context, fall back to the provider/model base rate when one exists.
- If only some token categories are priced, estimate those and leave the rest unpriced.
- If the exact model is unknown but the provider model family is known, allow an opt-in family-level estimate.
- Every estimate should carry provenance and confidence metadata.

## Gaps Before Accurate Cross-Provider Costing

The current observability labels are missing pricing context needed for several provider rules:

- service tier or execution mode (`standard`, `batch`, `priority`, `flex`, `fast`)
- data residency or billing region
- cache write TTL
- whether long-context pricing was activated
- counts for provider-side billed tools/searches/sessions
- actual billed model id when `responseModel` differs from request `model`

Without some of that context, we can only safely emit:

- base token cost estimates
- or provider-specific best-effort estimates with explicit caveats

For this product, that is acceptable as long as the result clearly says what assumptions were used.

## Distribution Model

Suggested packaging model:

1. `@mastra/costing` ships with an embedded recent pricing snapshot.
2. The package can also load a local snapshot file for offline overrides.
3. The package can optionally query a hosted pricing API for fresher data.
4. The caller chooses freshness strategy, for example:
   - `embedded-only`
   - `local-preferred`
   - `remote-preferred`
   - `remote-with-embedded-fallback`

This makes local development simple while allowing centrally improved pricing data later.

Updated implementation direction from later discussion:

- first pass should ship only `pricing_rules`
- we do not need the full base/delta publication model immediately
- the package can remain a thin estimator wrapper over a locally cached pricing dataset
- long term, distribution should happen through object storage/CDN rather than daily npm publishes

To avoid shipping the full rule list every day, distribution should be delta-based:

- ship a compact baseline snapshot with the package
- publish daily change sets that contain only new rules, rule closures, and corrections
- materialize the current effective rule set by applying deltas on top of the baseline

That implies the normalized format needs stable rule ids, temporal validity fields, and explicit correction semantics.

## Snapshot Strategy

Recommended artifact types:

- `base snapshot`: full effective rule set at a point in time
- `delta snapshot`: only `insert`, `close`, `correct`, and rare `delete` changes since the previous published snapshot
- `manifest`: metadata telling clients which base snapshot and deltas are needed

Practical model:

1. Generate a full normalized view internally for each capture date.
2. Diff it against the previous effective view.
3. Publish only the diff as the daily artifact.
4. Periodically publish a new compacted base snapshot so clients do not need an unbounded delta chain.

For example:

- package ships `base@2026-03-17`
- hosted service later publishes `delta@2026-03-18`, `delta@2026-03-19`, `delta@2026-03-20`
- a later release or compaction job publishes `base@2026-04-01`

This keeps package size and update bandwidth under control.

First-pass simplification:

- publish a full `pricing_rules` snapshot only
- defer daily delta publication until the normalized schema is stable
- keep the schema compatible with future base/delta updates, but do not implement them first

## Historical Semantics

The core rule is:

- price changes should preserve history
- corrections should repair bad history

For normal pricing evolution:

- do not overwrite old prices
- keep the old rule with its original `effective_start`
- set its `effective_end` when the provider price changes
- insert a new rule with the new price and a new `effective_start`

That allows recomputation to use the price that was effective at the time the model query happened.

For incorrect data, use correction semantics:

- if we parsed or published a wrong rule, emit a `correct` event or a rare `delete`
- corrections should be auditable and should explain why history changed
- recomputation should use corrected history once the correction is installed

Operationally, `delete` should be uncommon. Most legitimate provider changes are temporal transitions, not removals.

## Hosted API Direction

The hosted API does not need to do heavy computation at first. It can start as a snapshot distribution service.

Possible endpoints:

- `GET /pricing/latest`
- `GET /pricing/snapshots/:date`
- `GET /pricing/providers/:provider/models/:model`
- `POST /estimates`

Near-term recommendation:

- Start with snapshot fetch APIs.
- Keep estimation logic in the package so local and remote behavior stay consistent.
- Add a hosted estimation endpoint later if central execution becomes useful.

Refined direction:

- the remote system can start as static object storage plus a tiny manifest, not a full API
- the package should download and cache a pricing snapshot, then estimate locally
- remote request-time dependency should not be required for normal estimation

Minimal remote artifacts:

- `latest.json`
- `pricing_rules.parquet`

Possible future artifacts:

- `pricing_rules.jsonl` for debug/review
- base snapshots plus deltas once update volume justifies it

## Local Update Direction

Long term, users should be able to re-run estimates against newer pricing data without regenerating all original traces.

That implies:

- estimates should be reproducible against a named pricing snapshot
- usage data should remain stored independently from pricing data
- recalculation should be a pure join of usage records against a newer snapshot

This strongly favors versioned immutable pricing snapshots over in-place mutation.

For local updates, the simplest flow is:

1. load embedded base snapshot
2. optionally apply locally cached delta snapshots
3. estimate against the resulting effective ruleset

If users want a single-file local cache, we can always compact base + deltas into a new local base artifact after sync.

Current preferred local model:

- use DuckDB locally to query the cached pricing snapshot
- refresh from remote on a policy like daily or when stale
- fall back to an embedded snapshot when remote fetch is unavailable

This implies the local library should be a thin wrapper around:

1. snapshot discovery
2. local cache management
3. DuckDB-backed rule lookup
4. estimate computation over the resolved ruleset

## Early Recommendation

Treat the pricing system as two layers:

- Layer 1: provider pricing history
- Layer 2: runtime cost estimation from usage metrics

That separation matters because pricing history can be modeled now, while runtime cost accuracy depends on adding missing observability labels later.

Also treat estimate generation as deterministic over:

- a usage record
- a pricing snapshot
- a fallback policy

That will make later recalculation and debugging practical.

Internally we can still compute a full effective ruleset for any given date, but external distribution should default to compact bases plus deltas.

For historical recomputation, the relevant join is not "latest price" but "price whose effective interval contains the usage timestamp".
