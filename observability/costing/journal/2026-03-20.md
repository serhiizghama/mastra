  - stale values
  - naming mismatch
  - tier mismatch
  - cache-price mismatch

- Which non-provider source is the best fallback when provider data is unavailable?
- Which providers need explicit model-name aliasing before a rollup can be trusted?

## Rollup Direction

Current working direction for the initial rollup:

- use non-provider sources first to build the widest possible provider/model inventory
- then use provider-specific sources to override fallback pricing wherever official provider data exists

This suggests a two-layer approach.

### 1. Discovery Layer

Goal:

- maximize provider/model coverage

Current proposed source order:

- start with `models-dev`
- add provider/model entries missing from `price-per-token`
- then add anything still missing from `helicone`
- then add anything still missing from `open-router`
- use `langfuse` last, primarily for alias coverage rather than canonical baseline rows

Working interpretation:

- `models-dev` gives the best starting breadth
- `price-per-token` is the strongest next supplement
- `helicone` adds more long-tail model coverage
- `open-router` adds useful provider-channel variation
- `langfuse` helps most with aliasing and pattern matching

### 2. Authority Layer

Goal:

- use the best available price for each provider/model/pricing-parameter combination

Current proposed rule:

- if provider-specific pricing exists, prefer it over any non-provider source
- if provider-specific pricing does not exist, keep the best available non-provider fallback

This keeps breadth and price authority separate:

- aggregators define inventory coverage
- provider-specific sources define pricing truth where available

### Pricing Shape

The initial rollup should not flatten all pricing into one row per model.

The working unit should preserve at least:

- `provider`
- `model`
- `meter`
- `service_tier`
- `pricing_mode`
- cache role such as read vs write
- threshold split where applicable

Examples of pricing dimensions that should remain explicit:

- OpenAI `standard`, `batch`, `flex`, `priority`
- Google long-context threshold splits
- Google `output_reasoning_tokens`
- cache read vs cache write pricing

### Working Principle

- widest list first
- then provider-specific override
- then parameter-specific pricing detail

This should let us build:

- a broad model/provider inventory
- a provider-authoritative pricing layer
- a fallback path for uncovered providers and models
