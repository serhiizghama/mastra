## Findings

### 1. The dashboard foundation should be token-first

There is no real cost metric yet, so the initial dashboard foundation should be:

- token volume
- request counts
- token trends
- token composition
- entity/provider/model breakdowns

Costing should layer on top of that, not replace it.

### 2. Request count already has a useful proxy

`count(mastra_model_total_input_tokens)` is a reasonable model-request proxy because the top-level input metric is emitted for every model generation with usage.

### 3. The frontend MVP already exposes the real backend gaps

The frontend MVP depends on:

- `groupBy: ['model']`
- `groupBy: ['status']`
- `groupBy: ['entityName', 'status']`

So the group-by issue is not theoretical. It is already a concrete backend mismatch.

### 4. DuckDB should support label-aware `groupBy`

Even after promoting some canonical dimensions to columns, label-aware `groupBy` in DuckDB still looks necessary:

- for backend consistency
- for future flexible dimensions
- to avoid over-promoting every label into a column

### 5. Canonical dashboard dimensions belong in columns

The exploration converged on promoting stable product-level dimensions into first-class metric columns:

- `status`
- `provider`
- `model`
- `environment`
- `serviceName`

Current thought:

- stored metric rows should treat these columns as canonical
- they should not need to remain duplicated in stored `labels`

### 6. New pricing-only context generally belongs in metadata

Pricing-calculation details and provenance do not need to become core dashboard dimensions by default.

Good fits for `metadata`:

- pricing source
- pricing version/snapshot ID
- cost computed timestamp
- rule-explanation details
- other pricing-specific debug fields

### 7. Cost JSON should not be the primary analytical representation

A JSON cost blob is fine for provenance context, but it is a poor primary shape for:

- aggregation
- grouping
- filtering
- backend consistency

### 8. Cost on token rows looks like the strongest MVP path

Current preferred direction:

- store estimated cost on the existing token-related metric rows
- accept that stored estimates may become stale over time
- prefer simplicity over a more complex recomputation/versioning system in v1

Important semantic rule:

- the cost on a metric row must be the cost for that row only
- do not copy full request-total cost across multiple sibling rows

### 9. The metric row is becoming multi-measure

The row model is trending toward:

- usage in `value`
- cost in `estimatedCost`

That implies the OLAP query layer should support measure selection and likely multi-measure querying.

### 11. Per request should mean per trace

For Mastra product semantics:

- one trace = one request
- per-request cost should be modeled per trace
- per-request token usage should be modeled per trace

This is the right user-facing unit even though a trace can contain multiple internal provider/tool calls.

## Decisions

- Treat this file as the primary record for today’s discussion, research, findings, and decisions
- Ignore documentation and changesets for the remainder of this exploration unless explicitly requested
- For the first metrics dashboard iteration, plan around token usage queries only
- Treat request count as `count(mastra_model_total_input_tokens)` for the initial dashboard
- Prefer entity/time slices first because those groupings are already compatible with the current backend shape
- Treat recomputation/backfill ability as a first-class evaluation criterion when comparing costing strategies
- Default new pricing-context fields toward `metadata`, not `labels`
- Treat optional first-class OLAP columns as the preferred home for canonical cross-backend dashboard dimensions
- Lean toward stored estimated cost on token-related metric rows for the initial product, accepting some staleness in exchange for a simpler design
- `estimatedCost` and `costUnit` should be first-class metric columns
- Most cost-calculation/provenance details should live in `metadata`
- Canonical dimensions like `status`, `provider`, and `model` should become first-class metric columns
- `environment` should be a first-class metric column
- `serviceName` should remain a first-class metric column since it already exists in the schema
- Both `environment` and `serviceName` need reliable propagation through the metrics emission/storage path for token metrics
- DuckDB should gain label-aware `groupBy` support now, even if some canonical dimensions are promoted to columns
- For Mastra dashboard semantics, “per request” should default to “per trace”

## Current Convergence Summary

- Token usage remains the primary observability signal for the dashboard foundation
- Estimated cost is trending toward a simple on-row storage model rather than a separate derived-cost persistence system
- The metric row model is becoming multi-measure:
  - usage in `value`
  - cost in `estimatedCost`
- Cost amount and cost unit should be first-class columns
- Most pricing calculation details should live in `metadata`
- Canonical dashboard dimensions should be promoted to first-class columns
- Remaining flexible dimensions should still be supported via label-aware `groupBy` in DuckDB
- Per-request dashboard metrics should be modeled per trace
